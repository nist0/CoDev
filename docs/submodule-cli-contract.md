# CoDev Submodule — CLI Contract

> Canonical reference for `codev init`, `codev update`, and `codev teardown`.
> All implementation must conform to this contract. The `codev.json` schema
> is the machine-readable counterpart at `schemas/codev.schema.json`.

---

## Commands

### `codev init`

Wire the CoDev submodule into the current repository.

```text
codev init [--strategy <extend|override>] [--overrides-dir <path>]
```

**Flags**

| Flag | Default | Description |
| --- | --- | --- |
| `--strategy` | `extend` | Merge strategy: `extend` appends host assets alongside submodule assets; `override` replaces a managed asset when a same-named file exists in `overridesDir` |
| `--overrides-dir` | `codev-overrides/` | Directory where host-specific customizations live (never managed by CoDev) |
| `--submodule-path` | `tools/codev` | Override the submodule path (defaults to the value in `codev.json` or `tools/codev` on first init) |

**What it does**

1. Reads (or creates) `codev.json` at the repository root.
2. Creates real directories at `.github/agents/`, `.github/prompts/`, `.github/instructions/`, and `.github/skills/`. In **symlink mode** (preferred), wires each CoDev agent, prompt, and instruction as an individual file symlink inside the corresponding real directory, leaving any pre-existing host files untouched. Skill themes are wired as theme-level directory symlinks inside `.github/skills/`. In **lockfile mode** (fallback, Windows without Developer Mode, or WSL on `/mnt/<drive>/...`), individual files are copied rather than symlinked, and `codev-lock.json` is written. Aborts with exit code 3 if a host file has the same name as a CoDev file.
3. Generates `copilot-instructions.md` via the manifest merge script.
4. Adds managed paths to `.gitignore` (symlink mode) or writes `codev-lock.json` (lockfile mode).
5. Installs the pre-commit hook at `.git/hooks/pre-commit`.
6. Prints a summary: mode used, managed paths, override strategy, and verification command.

**Exit codes**

| Code | Meaning |
| --- | --- |
| 0 | Success |
| 1 | `codev.json` validation failed |
| 2 | Submodule path does not exist or is not a valid CoDev submodule |
| 3 | Partial state left; teardown recommended |

### `codev update`

Re-sync host `.github/` after `git submodule update`.

```text
codev update
```

**What it does**

1. Reads `codev.json`.
2. In symlink mode: symlinks are already live; re-generates `copilot-instructions.md` only.
3. In lockfile mode: re-copies changed managed files; updates `codev-lock.json` SHA256 entries; re-runs manifest merge.
4. Validates `codev.json` against the JSON Schema.
5. Prints a diff summary of changed files.

Must be run after every `git submodule update --remote` in lockfile mode. In symlink mode this is a no-op for assets (symlinks follow the submodule automatically) but should still be run to refresh the generated manifest.

### `codev teardown`

Fully remove all CoDev-managed assets and return the host repo to pre-init state.

```text
codev teardown [--force]
```

**Flags**

| Flag | Default | Description |
| --- | --- | --- |
| `--force` | false | Skip confirmation prompt |

> **Windows / non-interactive shells**: pass `--force`. Without it the CLI waits
> for a `y` / `N` keystroke; PowerShell cannot forward this when the script is
> invoked indirectly (e.g. via a task runner or CI step), causing the process to
> appear frozen until killed.

**What it does**

1. Removes all symlinks (symlink mode) or all copied managed files + `codev-lock.json` (lockfile mode).
2. Removes `.git/hooks/pre-commit` installed by CoDev (leaves it if it was already present before init).
3. Removes CoDev-added lines from `.gitignore`.
4. Removes `copilot-instructions.md` if it was generated by CoDev.
5. Does **not** touch `codev.json`, `codev-overrides/`, or any file authored by the host.
6. Prints teardown summary and confirms clean state.

**Guaranteed safe**: host-authored files in `codev-overrides/` are never deleted.

## `codev.json` — Manifest

Placed at the repository root. Validated against `schemas/codev.schema.json` on every `init` and `update`.

### Example

```json
{
  "$schema": "https://raw.githubusercontent.com/nist0/CoDev/main/schemas/codev.schema.json",
  "version": "1",
  "submodulePath": "tools/codev",
  "overrideStrategy": "extend",
  "overridesDir": "codev-overrides",
  "pinnedRef": "v1.2.0",
  "managedPaths": [
    ".github/agents",
    ".github/skills",
    ".github/prompts",
    ".github/instructions",
    ".github/copilot-instructions.md"
  ]
}
```

### Fields

| Field | Type | Required | Default | Description |
| --- | --- | --- | --- | --- |
| `$schema` | string | No | — | JSON Schema URI for editor validation |
| `version` | string | Yes | — | Manifest schema version (currently `"1"`) |
| `submodulePath` | string | Yes | — | Relative path from repo root to the CoDev submodule directory |
| `overrideStrategy` | `"extend"` or `"override"` | Yes | — | How host assets interact with submodule assets |
| `overridesDir` | string | No | `"codev-overrides"` | Relative path to host customization directory |
| `pinnedRef` | string | No | — | Git tag or commit SHA for documentation/audit; does not change submodule state |
| `managedPaths` | array of strings | No | All CoDev `.github/` asset paths | Explicit list of paths managed by CoDev in this repo |

## Override strategy semantics

### `extend` (default)

- Submodule assets are present at their normal paths.
- Host adds **new** files (new agents, new skills, etc.) in `codev-overrides/`, which are symlinked/copied alongside submodule assets.
- No submodule-managed file is replaced.
- Any file with the same name as a submodule-managed file in `codev-overrides/` triggers a **warning** (not an error) at init time.

### `override`

- Submodule assets are present at their normal paths.
- Host files in `codev-overrides/` with the **same relative name** as a submodule-managed asset **shadow** the submodule version.
- VS Code Copilot loads the host version; the submodule version is masked.
- Masked files are listed in `codev-lock.json` under `"overrides"` for auditability.

## `codev-overrides/` directory convention

```text
codev-overrides/
  agents/          # host-specific agents (extend) or override agents (override strategy)
  skills/          # host-specific skills
  prompts/         # host-specific prompts
  instructions/    # host-specific instructions
  copilot-instructions.override.md   # host section appended to / replacing the generated manifest
```

Files in `codev-overrides/` are **never touched** by `codev update` or `codev teardown`.
This directory should be committed to the host repository.

## Windows Developer Mode detection

CoDev detects symlink availability at init time:

```powershell
# PowerShell detection logic
$devMode = (Get-ItemProperty `
  "HKLM:\SOFTWARE\Microsoft\Windows\CurrentVersion\AppModelUnlock" `
  -ErrorAction SilentlyContinue).AllowDevelopmentWithoutDevLicense -eq 1

if (-not $devMode) {
    Write-Warning "Windows Developer Mode not enabled. Falling back to lockfile mode."
    Write-Warning "To use symlink mode: Settings > Update & Security > For developers > Developer Mode."
}
```

When run from WSL against a repo on `/mnt/<drive>/...`, CoDev also falls back to
lockfile mode. This avoids creating Linux-target symlinks such as `/mnt/c/...`
inside a Windows repository, which Windows tools do not resolve as proper links.

In lockfile mode, all behaviour is identical from the consumer's perspective; the only difference is that `codev update` must be run after `git submodule update`.

## Managed-file invariant

> **No file in a managed path may be edited directly in the host repository.**

- The pre-commit hook enforces this locally.
- The CI integrity workflow enforces this in every PR.
- To change a managed asset: open a PR against the CoDev submodule repository, then update the submodule reference in the host repo.
- To layer on top of a managed asset: use `codev-overrides/` with the `extend` or `override` strategy.
